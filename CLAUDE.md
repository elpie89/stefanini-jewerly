This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**Showcase-only** site for a friend's jewelry shop (Stefanini). Built with **Jekyll**, deployed to **GitHub Pages**. No e-commerce, no cart, no product catalog page, no blog. Originally started from the purchased "Mojuri" HTML template (index5 "Home Parallax" variant); all shop/blog/account UI has been removed.

- Repo: `git@github.com:elpie89/stefanini-jewerly.git` (note: the repo name has an extra N vs the local folder `stefani-jewerly`).
- Deployed URL: `https://elpie89.github.io/stefanini-jewerly/` → `_config.yml` sets `baseurl: /stefanini-jewerly`.
- Site language: English.

## Commands

### Local dev
```bash
bundle install           # first time, installs jekyll + webrick from Gemfile
bundle exec jekyll serve # http://localhost:4000/stefanini-jewerly/
```

Local dev serves with the configured `baseurl`, so the local URL includes `/stefanini-jewerly/` too. To preview without the baseurl locally, run `bundle exec jekyll serve --baseurl ""`.

### Production build
Handled by [.github/workflows/jekyll.yml](.github/workflows/jekyll.yml) on every push to `main` — runs `jekyll build` with the baseurl auto-detected from the GitHub Pages environment, then deploys via `actions/deploy-pages@v4`. No manual build needed.

## Architecture

### Pages
Only four pages exist at the repo root:

- [index.html](index.html) — home (Home Parallax layout from index5: four full-screen parallax hero blocks).
- [about.html](about.html) — About Us.
- [contact.html](contact.html) — Contact (map iframe + info + form).
- [404.html](404.html) — served by GitHub Pages when a URL doesn't match.

All four have Jekyll front matter (`layout: default`, plus per-page `body_class`, `header_class`, `header_logo`, `show_title_bar`, `title`, `description`). Front matter drives the shared layout's per-page variations.

### Shared chrome
[_layouts/default.html](_layouts/default.html) wraps every page in the template's page-wrapper and pulls in four includes:

- [_includes/head.html](_includes/head.html) — `<head>`: all CSS (Bootstrap, icon fonts, Slick, mmenu, `app.css`, `responsive.css`), Google Fonts, favicon, dynamic `<title>`/`<description>` from front matter.
- [_includes/header.html](_includes/header.html) — site header with the 3-item menu (Home / About / Contact). **Cart, wishlist, login, search, and the mega-menu have been deliberately removed.** The mobile menu is auto-generated by [assets/js/app.js:143](assets/js/app.js#L143) cloning `#main-navigation` — so a single nav definition serves both desktop and mobile.
- [_includes/footer.html](_includes/footer.html) — centered logo + social icons + copyright. The template's original 3/4-column footer (Customer Services, About Us, shipping links) has been stripped.
- [_includes/scripts.html](_includes/scripts.html) — vendor JS (popper, jQuery, Bootstrap, Slick, mmenu) then `app.js`. Order matters — see below.

### Asset paths
All CSS/JS/image references in shared includes use `{{ '/path' | relative_url }}` so they resolve correctly under the `baseurl`. **Do not use raw absolute paths** (`/assets/...`) in includes — they skip the baseurl. In page content files, relative paths like `src="media/..."` also work, but prefer `relative_url` for consistency when editing.

### Vendor libraries ([libs/](libs/))
Vendored (not installed). In use site-wide: `bootstrap`, `jquery`, `popper`, `slick`, `mmenu`, icon fonts (`feather-font`, `icomoon-font`, `font-awesome`, `wpbingofont`, `elegant-icons`). The `elevatezoom`, `select2`, `slider` folders are unused by the current pages — safe to leave in `libs/` but they ship no-cost bytes.

### Script include order (matters)
[_includes/scripts.html](_includes/scripts.html) loads: popper → jquery → bootstrap → slick → mmenu → `app.js`. `app.js` is a jQuery IIFE that depends on all of the above; reordering breaks the site.

### Media ([media/](media/))
Template images (banners, brand logos, testimonials, product photos, favicon, logos). The home page references `bg-img-1` … `bg-img-4` as CSS classes defined in [assets/css/app.css](assets/css/app.css) — the actual image files are wired up through those classes, not direct `<img>` tags. Replacing hero imagery means editing the CSS background-image rules, not the HTML.

### Contact form
The form in [contact.html](contact.html) posts to an empty `action=""` — it does nothing when submitted. To make it actually send mail, sign up for Formspree (or similar) and replace the action with a Formspree endpoint. A comment above the `<form>` documents this.

## Things to avoid

- Do **not** reintroduce cart/wishlist/account/login UI, or shop/blog links. This is a vetrina, not a store.
- Do **not** add new `index*.html` variants — only `index.html` exists and it's the Home Parallax layout.
- Do **not** use `site.url` in `href=`/`src=` attributes — use `relative_url` (which applies `baseurl`). `absolute_url` prepends `site.url` and will produce broken links when testing locally or forking.
